# Vintage Hub 기획 설계

작성일: 2026-05-14

## 배경

빈티지 쇼핑몰은 사이트 수가 많고, 각 사이트의 검색 기능과 실측 검색 기능이 제한적이다. 사용자는 원하는 매물을 찾기 위해 여러 사이트를 반복 방문해야 하며, 상품 상세에 숨어 있는 실측 정보를 직접 확인해야 한다.

Vintage Hub는 여러 빈티지 쇼핑몰의 공개 상품 정보를 크롤링하고 정규화해, 한 곳에서 최신 매물을 검색하고 실측 기준으로 필터링할 수 있게 하는 서비스다.

## MVP 범위

MVP의 핵심 가치는 다음 세 가지다.

- 여러 빈티지 사이트의 신상품을 최신 크롤링 순으로 모아보기
- 사이트마다 불편한 검색을 `사이트`, `카테고리`, `실측`, `가격`, `품절여부` 필터로 통합하기
- 상품 상세에서 실측, 상세 설명, 원본 상품 URL을 확인하기

MVP에서 제외하는 기능은 다음과 같다.

- 일반 사용자 로그인
- 관심 상품 저장
- 검색 조건 알림
- 결제 또는 구매 대행
- 이미지 OCR 기반 실측 추출
- Elasticsearch/OpenSearch 즉시 도입

## 화면 구조

공개 사용자 화면과 운영자 화면을 분리한다.

### 공개 사용자 화면

공개 사용자 화면은 로그인 없이 접근 가능하다.

#### 메인 리스트

메인 화면은 최신 크롤링 순으로 상품을 보여준다. 상품 카드에는 다음 정보만 노출한다.

- 상품명
- 가격
- 대표 이미지
- 사이트
- 품절여부

메인 검색 옵션은 다음과 같다.

- 사이트
- 표준 대분류
- 표준 소분류
- 실측
- 가격
- 품절여부

실측과 상세 내용은 메인 카드에 노출하지 않고 상품 상세 화면에서 보여준다.

#### 상품 상세

상품 상세 화면에는 다음 정보를 보여준다.

- 상품명
- 실측
- 상세 내용
- 대표 이미지
- 실제 상품 상세 URL
- 품절여부

사용자가 상세 화면에 진입하면 백그라운드에서 TTL 정책에 따라 품절 여부를 재확인한다.

### 운영자 화면

운영자 화면은 URL 직접 접근만 허용한다. 공개 화면에는 관리자 로그인 버튼을 두지 않는다.

운영자는 사전 발급된 계정으로만 로그인할 수 있으며, 회원가입 기능은 제공하지 않는다. 운영자 화면에서는 다음 작업을 수행한다.

- 사이트별 크롤링 주기 설정
- 크롤러 상태 확인
- 파싱 실패 확인
- 리뉴얼/폐쇄 의심 상태 확인

운영자 인증은 JWT 기반으로 구현한다. 운영자 계정은 회원가입으로 만들지 않고, 암호화된 비밀번호를 미리 생성해 DB에 등록한다.

## 데이터 모델

핵심 엔티티는 다음과 같다.

- `CrawlSite`: 크롤링 대상 사이트
- `Product`: 정규화된 상품
- `ProductMeasurement`: 상품 실측
- `ProductCategoryMapping`: 원본 카테고리와 표준 카테고리 매핑
- `CrawlRun`: 크롤링 실행 기록
- `CrawlIssue`: 리뉴얼, 폐쇄, 차단, 파싱 실패 이슈
- `AdminUser`: 운영자 계정

### AdminUser

운영자 계정은 사전 발급 방식으로 생성한다. 비밀번호는 평문으로 저장하지 않고, 서버가 사용하는 비밀번호 해시 정책으로 암호화해 저장한다.

운영자 로그인 성공 시 서버는 JWT를 발급한다. 운영자 API는 JWT 인증이 통과된 요청만 허용한다.

### Product

상품에는 다음 필드를 둔다.

- 상품명
- 원가
- 할인가
- 품절 상태
- 상품 상세 내용
- 상품 상세 URL
- 대표 이미지 URL
- 사이트
- 원본 상품 식별자
- 원본 카테고리
- 표준 대분류
- 표준 소분류
- 카테고리 분류 신뢰도
- 수집 시각
- 마지막 확인 시각
- 품절 확인 시각

원가/할인가를 모두 제공하지 않는 사이트에서는 확인된 가격을 원가에 저장하고 할인가는 `null`로 둔다. 상품 가격 자체를 확인할 수 없는 경우 원가와 할인가를 모두 `null`로 둔다.

품절 상태는 boolean이 아니라 상태값으로 관리한다.

- `AVAILABLE`
- `SOLD_OUT`
- `UNKNOWN`
- `CHECK_FAILED`

### ProductMeasurement

실측은 상품 테이블의 고정 컬럼으로 두지 않고 별도 테이블로 둔다. 필드는 다음과 같다.

- `part`
- `valueCm`
- `rawText`
- `confidence`
- `source`
- `updatedAt`

`source`는 `AUTO`, `MANUAL` 같은 값으로 자동 추출과 운영자 보정을 구분한다.

## 카테고리 정책

사이트별 카테고리 기준이 다르고, 카테고리가 없는 상품도 있으므로 원본 카테고리와 서비스 표준 카테고리를 분리한다.

상품에는 다음 카테고리 정보를 저장한다.

- `sourceCategoryName`: 원본 사이트 카테고리명
- `standardCategory`: 표준 대분류
- `standardSubCategory`: 표준 소분류
- `categoryConfidence`: 자동 분류 신뢰도
- `categorySource`: 분류 근거

표준 대분류는 다음으로 시작한다.

- `상의`
- `하의`
- `아우터`
- `신발`
- `가방`
- `액세서리`
- `기타`
- `미분류`

표준 소분류는 다음으로 시작한다.

- `데님`
- `니트`
- `셔츠`
- `스웻`
- `자켓`
- `코트`
- `팬츠`
- `스커트`

카테고리가 없는 상품은 기본적으로 `미분류`로 둔다. 상품명, 상세 설명, 실측 키워드로 추론 가능한 경우에만 자동 분류하고, 확신이 낮으면 운영자 보정 대상으로 남긴다.

## 크롤링 정책

크롤링은 사이트별 설정 가능한 주기로 실행한다. 기본 목표는 상품 변경을 1-3시간 안에 반영하는 것이다.

초기 크롤링 대상 우선순위는 로켓샐러드를 최우선으로 한다. 그 이후 사이트 우선순위는 크롤링 안정성, 상품 수, 파싱 난이도를 확인한 뒤 별도로 결정한다.

운영자는 사이트별로 다음 주기 중 하나를 설정할 수 있다.

- 30분
- 1시간
- 3시간
- 6시간
- 12시간
- 24시간

매 실행마다 전체 상세를 바로 수집하지 않는다. 먼저 목록 페이지나 sitemap의 fingerprint를 비교한다. 변경이 감지되면 신규/변경 상품 상세를 수집하고, 기존 상품은 TTL 기준으로 재확인한다.

사이트별 크롤링 운영 필드는 다음을 둔다.

- `crawlIntervalMinutes`
- `lastCrawledAt`
- `lastChangedAt`
- `lastChangeDetectedAt`
- `consecutiveNoChangeCount`
- `crawlerStatus`

크롤러 상태는 다음으로 관리한다.

- `ACTIVE`
- `PAUSED`
- `ERROR`
- `BLOCKED`
- `RENEWAL_SUSPECTED`
- `CLOSED_SUSPECTED`

## 품절 갱신 정책

품절 여부는 TTL 기반 혼합 정책으로 갱신한다.

- 메인 리스트는 DB의 현재 품절 상태를 표시한다.
- 상품 상세 진입 시 백그라운드로 품절 여부를 재확인한다.
- 최근 확인 시간이 TTL 안이면 원본 사이트 호출을 생략한다.
- 판매중 상품은 기본 30분 TTL을 둔다.
- `UNKNOWN`, `CHECK_FAILED` 상품은 상세 진입 시 즉시 재확인하고, 실패가 반복되면 5분 TTL을 둔다.
- 이미 품절 확정 상품은 24시간 TTL을 둔다.
- 결과가 바뀌면 DB를 업데이트하고 클라이언트에 이벤트를 전송한다.

품절 확인 운영 필드는 다음을 둔다.

- `availabilityCheckedAt`
- `availabilityCheckFailCount`
- `availabilitySource`

MVP에서는 WebSocket 대신 SSE를 사용한다. 품절 상태 변경처럼 서버에서 클라이언트로 보내는 단방향 이벤트에는 SSE가 충분하며 구현과 운영이 단순하다.

## 리뉴얼/폐쇄 감지

리뉴얼/폐쇄 감지는 자동 확정이 아니라 의심 상태로 관리한다.

MVP에서 사용하는 감지 신호는 다음과 같다.

- 응답 코드 변화
- 연속 실패 횟수
- HTML 크기 급변
- 상품 링크 수 급감
- 주요 selector 누락
- 파서 실패율 증가

예를 들어 `200`이던 사이트가 반복적으로 `404`, `403`, `429`, `5xx`를 반환하면 이슈를 만든다. 목록에서 상품 URL이 갑자기 0개가 되거나, 기존 selector로 상품명/가격/이미지를 찾지 못하는 비율이 증가해도 리뉴얼 의심으로 표시한다.

## 실측 파싱 정책

실측 파싱은 텍스트 기반 자동 추출과 운영자 보정을 조합한다.

- 상세 HTML에서 텍스트를 추출한다.
- 정규식과 규칙 기반으로 실측 후보를 파싱한다.
- 부위명과 단위를 표준화한다.
- 파싱 신뢰도를 저장한다.
- 실패하거나 신뢰도가 낮은 상품은 운영자가 보정한다.
- 상세 원문은 그대로 저장해 파서 개선에 활용한다.

대표 실측 필드는 다음과 같다.

- 상의: `어깨`, `가슴`, `소매`, `총장`
- 하의: `허리`, `밑위`, `허벅지`, `밑단`, `총장`
- 아우터: `어깨`, `가슴`, `소매`, `총장`

`cm`, `inch`, `단면`, `약`, `오차` 같은 표현은 표준 단위와 신뢰도 정보로 정규화한다.

## 검색 정책

검색은 MVP에서 PostgreSQL 기반 조건 검색으로 시작한다. 검색 엔진은 바로 도입하지 않고, 상품 수나 검색 품질 문제가 실제로 생긴 뒤 Elasticsearch/OpenSearch로 분리한다.

MVP 검색 대상은 다음이다.

- 상품명 텍스트
- 사이트
- 표준 대분류
- 표준 소분류
- 실측 범위
- 가격 범위
- 품절 상태
- 최신 크롤링 순 정렬

상세 내용은 저장하지만 MVP 텍스트 검색 대상에서는 제외한다. MVP의 텍스트 검색은 상품명을 기준으로 하고, 상세 내용 검색, 동의어, 브랜드명, 영문/한글 혼용, 오타 검색은 검색 엔진 도입 시점에 고도화한다.

## MVP 이후 확장 후보

- 사용자 계정
- 관심 상품 저장
- 검색 조건 저장
- 새 상품 알림
- 이미지 OCR 기반 실측 추출
- Elasticsearch/OpenSearch 검색 엔진
- 브랜드/스타일 기반 자동 태깅
- 파서 실패 케이스 기반 자동 개선 도구
- 카테고리/실측 운영자 보정 UI

### 카테고리/실측 운영자 보정 UI

카테고리/실측 보정 UI는 자동 수집과 자동 파싱이 실패하거나 신뢰도가 낮은 상품을 운영자가 최소한의 조작으로 바로잡기 위한 추후 개발 기능이다. 전체 상품 편집기가 아니라 `검색 품질에 직접 영향을 주는 필드`만 수정할 수 있게 한다.

보정 대상은 다음이다.

- 표준 카테고리가 `미분류`인 상품
- 카테고리 분류 신뢰도가 낮은 상품
- 실측 파싱에 실패한 상품
- 실측 파싱 신뢰도가 낮은 상품
- 파서 오류가 발생한 상품

카테고리 보정에서는 표준 대분류, 표준 소분류, 보정 메모만 수정한다. 운영자가 카테고리를 수정하면 `categorySource`는 `MANUAL`로 바꾸고, `categoryConfidence`는 `1.0`으로 저장한다. 원본 카테고리명은 수정하지 않는다.

실측 보정에서는 원본 상세 텍스트와 자동 추출 결과를 나란히 보여주고, 운영자가 자동 추출된 실측값을 수정하거나 누락된 실측값을 추가할 수 있게 한다. 운영자가 수정한 `ProductMeasurement.source`는 `MANUAL`로 저장하고, 자동 추출 원문인 `rawText`는 유지한다.

## 구현 전 추가 결정 사항

현재 설계 기준에서 구현 전 추가로 결정할 사항은 다음이다.

- 운영자 계정의 초기 ID, 비밀번호 생성 방식, 비밀번호 해시 알고리즘
- JWT access token 만료 시간과 refresh token 사용 여부
