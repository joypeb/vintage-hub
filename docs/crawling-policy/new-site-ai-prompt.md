# 신규 사이트 크롤링 분석 AI 프롬프트

이 문서는 신규 빈티지 쇼핑몰을 크롤링 대상으로 추가하기 전에, AI에게 사이트 분석과 매핑 정책 초안을 맡길 때 사용하는 표준 프롬프트다.

목표는 구현 코드를 바로 작성하는 것이 아니라 다음 결정을 먼저 명확히 하는 것이다.

- 어떤 URL과 페이지를 수집할지
- 어떤 HTML/API 위치에서 어떤 상품 필드를 가져올지
- 원본 카테고리를 Vintage Hub 표준 카테고리로 어떻게 매핑할지
- 상세 본문의 실측 정보를 어떤 규칙으로 추출할지
- robots.txt, 요청 간격, 차단 신호 등 운영 리스크를 어떻게 다룰지

## 사용 방법

1. `{SITE_NAME}`, `{SITE_URL}`, `{TARGET_URLS}` 값을 실제 사이트 정보로 바꾼다.
2. 가능하면 홈, 주요 카테고리, 상품 상세 URL을 각각 1개 이상 넣는다.
3. AI에게 웹 접근 권한이 있으면 실제 페이지를 확인하게 하고, 없으면 HTML 샘플을 함께 제공한다.
4. 결과는 `docs/crawling-policy/sites/{site-code}.md` 초안으로 저장할 수 있는 수준의 마크다운으로 받는다.

## 표준 프롬프트

```text
너는 Vintage Hub 백엔드 프로젝트의 신규 쇼핑몰 크롤링 정책을 설계하는 역할이다.

Vintage Hub는 여러 빈티지 쇼핑몰의 공개 상품 정보를 수집해 상품명, 가격, 품절 상태, 이미지, 원본 카테고리, 표준 카테고리, 실측 정보를 정규화한다.

대상 사이트:
- 사이트명: {SITE_NAME}
- 사이트 URL: {SITE_URL}
- 우선 확인할 URL:
{TARGET_URLS}

분석 원칙:
- robots.txt와 차단 경로를 먼저 확인한다.
- 로그인, CAPTCHA, 403, 429, robots.txt 차단이 있으면 우회 방법을 제안하지 말고 수집 제외 또는 공식 API/허가 기반 접근으로 판단한다.
- 공개 상품 목록과 공개 상품 상세에서 확인 가능한 정보만 수집 대상으로 삼는다.
- 브라우저 자동화가 꼭 필요한지, HTTP 요청과 HTML 파싱만으로 충분한지 구분한다.
- HTML에 포함된 JSON-LD, 메타 태그, 전역 JSON, data-* 속성, 상품 카드 DOM, 상세 본문 DOM을 우선순위별로 확인한다.
- 정렬, 검색, 필터, 장바구니, 주문, 로그인, 회원, 관리자 URL은 수집 대상에서 제외한다.
- 불확실한 값은 추정으로 저장하지 말고 "운영자 보정 필요" 또는 "수집하지 않음"으로 표시한다.

아래 항목을 반드시 분석해 마크다운으로 답하라.

## 1. 결론

- 크롤링 가능 여부: 가능 / 조건부 가능 / 비권장 또는 불가
- 권장 수집 방식: 정적 HTML / JSON API / 브라우저 렌더링 필요 / 공식 API 필요
- 우선 수집 대상 여부: 높음 / 보통 / 낮음
- 가장 큰 리스크 3개

## 2. 접근 정책

다음을 확인해 표로 정리하라.

- robots.txt URL
- 허용/차단 규칙
- 수집 가능한 목록 URL 패턴
- 수집 가능한 상세 URL 패턴
- 제외해야 할 URL 패턴
- 권장 요청 간격
- 권장 병렬 요청 수
- 필요한 인코딩 처리
- User-Agent 주의사항

## 3. 수집 시작점

신규/변경 상품을 안정적으로 찾기 위한 시작점을 정리하라.

- 홈 신상품 영역
- 주요 물리 카테고리 목록
- sitemap 사용 가능 여부
- 페이지네이션 방식
- 정렬 파라미터
- 중복 상품이 많은 브랜드/세일/기획전 카테고리

목록 URL은 최신순 또는 기본 정렬을 우선 사용한다. 가격순, 인기순, 검색 URL은 기본 수집 대상에서 제외한다.

## 4. 상품 필드 추출 정책

Vintage Hub가 저장하는 필드별로 "어디서", "어떤 우선순위로", "어떻게 정규화할지"를 표로 작성하라.

필수 필드:

| Vintage Hub 필드 | 의미 | 분석할 내용 |
|---|---|---|
| sourceProductId | 원본 사이트 상품 식별자 | URL query, path id, JSON id 중 무엇을 쓸지 |
| productName | 상품명 | 상세 JSON-LD, 상세 DOM, 목록 DOM 우선순위 |
| originalPrice | 원가 또는 판매가 | 숫자 정규화, 통화, 할인 구조 |
| salePrice | 할인가 | 없으면 null |
| stockStatus | 품절 상태 | InStock, OutOfStock, SOLD OUT DOM, 버튼 상태 |
| description | 상세 설명 | 텍스트 추출 위치, 이미지-only 상세 여부 |
| thumbnailImageUrl | 대표 이미지 | 절대 URL 정규화 방식 |
| productDetailUrl | 원본 상품 URL | 사용자가 클릭할 PC/모바일 원본 URL |
| sourceCategoryName | 원본 카테고리명 | 사이트 카테고리, breadcrumb, JSON-LD category |
| measurements | 실측 | 상세 본문에서 추출 가능한 부위와 패턴 |

추출 우선순위 예시:

1. 상세 JSON-LD Product 객체
2. 상세 페이지의 구조화된 DOM
3. og:title, og:image 같은 메타 태그
4. 목록 카드 DOM
5. 문서 title

가격은 숫자와 소수점을 제외한 문자를 제거해 KRW 숫자로 정규화한다. 값이 없거나 품절 상품에서 가격이 숨겨진 경우 null 허용 여부를 명시한다.

## 5. 표준 카테고리 매핑

원본 사이트 카테고리와 상품명/설명 키워드를 Vintage Hub 표준 카테고리로 매핑하라.

표준 대분류는 다음 값만 사용한다.

- 상의
- 하의
- 아우터
- 신발
- 가방
- 액세서리
- 기타
- 미분류

표준 소분류는 사이트에서 충분히 구분 가능한 경우에만 사용한다. 기본 후보는 다음이다.

- 데님
- 니트
- 셔츠
- 스웻
- 자켓
- 코트
- 팬츠
- 스커트
- 티셔츠
- 베스트
- 모자
- 벨트
- 쥬얼리
- 넥타이
- 기타

다음 형식의 매핑표를 작성하라.

| 원본 카테고리/키워드 | 표준 대분류 | 표준 소분류 | 신뢰도 | 근거 |
|---|---|---|---:|---|
| 예: TOP > SHIRT | 상의 | 셔츠 | 0.950 | 원본 카테고리가 직접 셔츠를 의미함 |

매핑 규칙:

- 원본 카테고리가 명확하면 신뢰도 0.950을 사용한다.
- 원본 카테고리는 대분류만 명확하고 소분류가 애매하면 신뢰도 0.800을 사용한다.
- 상품명/설명 키워드로만 추론하면 신뢰도 0.800 이하를 사용한다.
- 성별, 브랜드, 시즌, 세일, 추천 사이즈는 표준 카테고리로 쓰지 않는다.
- 확신이 낮으면 `미분류` 또는 null로 두고 운영자 보정 대상으로 표시한다.
- `sourceCategoryName`은 원본 문자열 그대로 저장하고, 표준 카테고리는 별도 필드로 저장한다.

## 6. 실측 추출 정책

상세 본문에서 자동 추출할 실측 정보를 분석하라.

대표 실측 필드는 다음을 우선 사용한다.

- 상의: 어깨, 가슴, 소매, 총장, 기장
- 하의: 허리, 밑위, 허벅지, 밑단, 총장, 기장
- 아우터: 어깨, 가슴, 소매, 총장, 기장
- 신발: 굽, 발볼, 총장 등 명확한 cm 값이 있을 때만
- 가방: 가로, 세로, 폭, 스트랩 등 명확한 cm 값이 있을 때만

실측 추출 규칙:

- `항목명 + 숫자 + cm` 패턴을 최우선으로 자동 추출한다.
- `44cm`, `44.5 cm`, `약 44cm`, `44 CM`은 cm 값으로 정규화한다.
- inch 값은 확실히 실측 단위로 쓰인 경우에만 cm 변환 여부를 제안하고, 추천 사이즈와 혼동되면 제외한다.
- `국내 100`, `33 인치 추천`, `(105)`, `M-L`, `Free`, `표기사이즈`는 실측으로 저장하지 않는다.
- 같은 항목이 여러 번 나오면 상세 본문의 실측 표기 영역 또는 `Size:` 이후에 가까운 값을 우선한다.
- 이미지에만 실측이 있고 텍스트가 없으면 OCR은 MVP 범위 밖으로 보고 "운영자 보정 필요"로 표시한다.
- 자동 추출 신뢰도가 낮으면 `ProductMeasurement`를 만들지 않는다.

다음 형식으로 실측 정책을 작성하라.

| 표준 부위명 | 원본 표기 후보 | 단위 | 자동 추출 여부 | 제외해야 할 혼동 표현 |
|---|---|---|---|---|
| 허리 | 허리, waist | cm | 가능 | 33인치 추천, W33 |

## 7. 품절 상태 판단

품절 상태를 다음 우선순위로 판단할 수 있는지 확인하라.

1. JSON-LD `offers.availability`
2. 상세 DOM의 SOLD OUT, 품절, sold-out 클래스
3. 구매 버튼 비활성화 또는 품절 버튼
4. 목록 카드의 SOLD OUT 배지

가능하면 다음 값으로 매핑하라.

- 판매중: AVAILABLE
- 품절: SOLD_OUT
- 판단 실패: UNKNOWN 또는 CHECK_FAILED

목록 품절 단서는 임시값으로만 사용하고, 상세 수집 결과로 확정할 수 있는지 명시하라.

## 8. 중복 제거와 URL 정규화

다음을 제안하라.

- `sourceProductId`로 사용할 값
- `siteCode + sourceProductId` 중복 제거 가능 여부
- query parameter 중 유지해야 할 값과 제거해야 할 값
- PC/모바일 URL 중 사용자에게 저장할 원본 URL
- 이미지 URL 절대경로 변환 규칙

## 9. 변경 감지와 운영 리스크

매 실행마다 전체 상세를 수집하지 않기 위한 변경 감지 방법을 제안하라.

- 목록 fingerprint 구성 값
- 신규 상품 판단 기준
- 기존 상품 품절 재확인 TTL
- 리뉴얼 의심 조건
- 폐쇄/차단 의심 조건

리뉴얼 의심 조건에는 최소한 다음을 포함하라.

- 주요 목록에서 상품 링크가 0개로 떨어짐
- 상세 JSON-LD 또는 핵심 selector가 사라짐
- 가격/품절/이미지 파싱 실패율이 급증함
- 기존 200 응답이 반복적으로 403, 404, 429, 5xx로 바뀜

## 10. 구현 초안

현재 Vintage Hub 크롤러 구조에 맞춰 구현 초안을 제안하라.

- siteCode 후보
- `SiteCrawler` 구현체 이름
- 목록 커서 형식
- 초기 커서 목록
- 목록 파서 selector/API 경로
- 상세 파서 selector/API 경로
- 카테고리 매핑 추가 위치
- 실측 정규식 후보
- 추가해야 할 테스트 케이스

구현 코드는 작성하지 말고, 파일 단위 작업 목록과 테스트 기준만 제안하라.

## 11. 최종 출력 형식

최종 답변은 `docs/crawling-policy/sites/{site-code}.md`에 바로 붙여 넣을 수 있는 마크다운으로 작성하라.

반드시 포함할 섹션:

- 결론
- 접근 정책
- 현재 확인된 값
- 목록 수집
- 상세 수집
- 카테고리 매핑 정책
- 실측 파싱 정책
- 품절 갱신 정책
- 중복 제거와 URL 정규화
- 변경 감지
- 운영 권장값
- 구현 시 확인할 테스트

각 주장에는 확인한 URL, selector, JSON key, 응답 상태, 또는 HTML 단서를 근거로 적어라. 근거가 없으면 추정이라고 표시하라.
```

## 결과 검토 체크리스트

AI가 작성한 결과는 구현 전에 사람이 다음 항목을 다시 확인한다.

- robots.txt 차단 경로를 수집 대상으로 넣지 않았는가
- 로그인/장바구니/주문/회원/관리자 URL이 제외되었는가
- 상품 식별자가 사이트 안에서 안정적인가
- 목록 수집만으로 중복 제거와 신규 상품 감지가 가능한가
- 상세 수집 없이 저장하면 안 되는 필드를 목록값만으로 확정하지 않았는가
- 가격, 품절, 이미지, 상세 URL의 추출 우선순위가 명확한가
- 원본 카테고리와 표준 카테고리를 혼동하지 않았는가
- 성별, 브랜드, 시즌, 세일 카테고리를 표준 의류 카테고리로 잘못 매핑하지 않았는가
- 추천 사이즈와 실측 cm 값을 혼동하지 않았는가
- OCR, 브라우저 자동화, 비공식 API 의존이 필요한 경우 MVP 범위와 운영 리스크를 분리해 적었는가
- 파서 실패율 증가, selector 누락, 403/429 등 리뉴얼/차단 신호가 정의되었는가
